# check-docs-sync スキル

永続ドキュメント（`docs/`）に書かれている内容と、実際のコードベースの状態を突き合わせ、乖離を検出して報告するスキルです。

## スキルの目的

- ドキュメントが「北極星」として機能し続けるよう、コードとのズレを早期に検出する
- 乖離の内容・重大度・修正方向（ドキュメントを直すべきか、コードを直すべきか）を明確にする
- スペック駆動開発の信頼性を維持する

## 使用タイミング

1. **定期チェック**: ユーザーが乖離チェックを依頼した時
2. **機能実装完了後**: `/add-feature` 完了後の健全性確認
3. **ドキュメント大幅更新後**: docs/ を書き換えた後の整合性確認
4. **リファクタリング後**: コード構造を変更した後

## チェックマトリクス

各ドキュメントに対して、以下の観点でコードと突き合わせる:

| ドキュメント | 突き合わせ対象 | チェック内容 |
|---|---|---|
| `docs/architecture.md` | `pyproject.toml`、ソースコード全体 | 技術スタック・依存パッケージ・Pythonバージョンが一致するか。DDD4層構造（domain / application / infrastructure / presentation）が実際のパッケージ構成に存在するか。依存方向ルール（domain層は何にも依存しない）が import 文レベルで守られているか |
| `docs/functional-design.md` | domain層・application層のコード | ドキュメント記載のエンティティ・値オブジェクト・集約・リポジトリがコードに存在するか。逆に、コードにあってドキュメントにないドメインモデルはないか。画面・API・機能の記載と実装が一致するか |
| `docs/repository-structure.md` | 実際のディレクトリツリー | 記載されたディレクトリ構造と実構造の差分。記載にないディレクトリ・ファイルの出現、記載にあるが存在しないパス |
| `docs/glossary.md` | コード内の命名（クラス名・関数名・変数名） | ユビキタス言語がコードの命名に反映されているか。用語集にない重要なドメイン概念名がコードに出現していないか |
| `docs/implementation-roadmap.md` | `.steering/` 配下の tasklist.md、実装済みコード | 「完了」とされた機能が実際に実装・テストされているか。実装済みなのにロードマップが未更新の機能はないか |
| `docs/development-guidelines.md` | テストコード、コーディングスタイル | テストの存在（TDD準拠）、記載された規約（命名規則・エラーハンドリング方針等）の遵守状況 |
| `docs/product-requirements.md` | 実装済み機能全体 | スコープ外とされた機能が実装されていないか（スコープクリープ検出）。※要求と実装の対応は functional-design 経由で確認するため、ここは概要レベルでよい |

## 実行手順

### ステップ1: 対象の確認

1. `docs/` 配下の正式版ドキュメントを Glob で列挙する（`docs/ideas/` は下書きのため**対象外**）
2. ソースコードの有無を確認する（`src/`、`tests/` 等を Glob で確認）
3. 判定:
   - **ドキュメントもコードも存在しない** → チェック不能。ユーザーに `/setup-project` を案内して終了
   - **ドキュメントのみ存在（コード未実装）** → 「未実装」は乖離ではない。implementation-roadmap.md 上で「完了」とされている項目のみチェックする
   - **両方存在** → ステップ2へ（フルチェック）

ユーザーが特定のドキュメントやディレクトリを指定した場合は、その範囲に限定する。

### ステップ2: コードベースの実態把握

サブエージェントに渡す前提情報として、以下を収集する:

```
1. ディレクトリツリー: Glob('**/*.py') 等でソース構成を取得
2. 依存関係: pyproject.toml を Read
3. テスト状況: Bash('uv run pytest --collect-only -q') でテスト一覧を取得
   （失敗してもチェックは継続。失敗自体を報告に含める）
4. 直近の変更: Bash('git log --oneline -20') と Bash('git diff HEAD --stat')
```

### ステップ3: 観点別チェックの並列実行

チェックマトリクスの各行について、**存在するドキュメントのみ**を対象に、general-purpose サブエージェントを**1つのメッセージで並列起動**する（独立した観点なので必ず並列にする）。

各サブエージェントへのプロンプトには以下を必ず含める:

```
[ドキュメントパス] を読み、[突き合わせ対象] と照合して乖離を検出してください。

チェック観点: [チェックマトリクスの該当行の内容]

各乖離について、以下の形式で報告してください:
- 乖離の内容（ドキュメントの記載箇所と、対応するコードの箇所を file:line 形式で明記）
- 重大度: [重大] / [軽微] / [要確認]（下記基準）
- 修正方向の推奨: 「ドキュメント更新」または「コード修正」と、その理由

重大度の基準:
- [重大]: 設計原則違反（レイヤー依存方向違反、ドメインモデルの不一致）、
  ドキュメントを信じて実装すると壊れるレベルの記載齟齬
- [軽微]: 命名の揺れ、ドキュメントの更新漏れ（構造変更が未反映等）
- [要確認]: どちらが正か判断できず、ユーザーの意図確認が必要なもの

乖離が無い場合は「乖離なし」と、確認した項目のリストと共に報告してください。
推測で乖離を報告しないこと。必ずファイルを読んで確認した事実のみ報告すること。
```

観点数が3つ以下の少規模チェックであれば、サブエージェントを使わず自分で直接 Read / Grep で確認してもよい。

### ステップ4: 結果の統合とレポート作成

全サブエージェントの結果を統合し、以下の形式でユーザーに報告する:

```markdown
# ドキュメント・コード乖離チェック結果

実施日: [日付]
チェック対象: [ドキュメント数] ドキュメント × コードベース

## サマリー

| 重大度 | 件数 |
|--------|------|
| [重大] | X件 |
| [軽微] | Y件 |
| [要確認] | Z件 |

## 検出された乖離

### [重大] 1. [タイトル]
- **ドキュメント**: [docs/xxx.md の記載内容]
- **コード**: [file:line の実態]
- **推奨対応**: [ドキュメント更新 or コード修正] — [理由]

（以下同様。重大度の高い順に列挙）

## 乖離なしと確認できた項目

- [項目リスト]

## 推奨アクション

1. [優先度順の対応提案]
```

重複する指摘は統合し、同一原因から派生した複数の乖離は1件にまとめて原因を明記する。

### ステップ5: 修正の提案（実施はユーザー承認後）

**このスキルの成果物はレポートである。検出した乖離を勝手に修正しないこと。**

- レポート提示後、対応するかどうかはユーザーの判断に委ねる
- ユーザーが修正を指示した場合:
  - **ドキュメント更新**: `docs/` 配下は1ファイルずつ更新し、ユーザーの承認を得てから次に進む（CLAUDE.md の原則に従う）
  - **コード修正**: `steering` スキルを使用してステアリングファイルを作成し、TDDサイクルで修正する（テストなしの修正は禁止）

## 注意事項

- **「未実装」と「乖離」を区別する**: implementation-roadmap.md で未着手・進行中とされている機能がコードに無いのは正常。乖離として報告しない
- **`docs/ideas/` は対象外**: 下書き・アイデアメモは正式仕様ではない
- **`.steering/` は履歴**: 過去のステアリングファイルと現在のコードのズレは原則報告不要。ただし implementation-roadmap.md の進捗確認の根拠としては参照する
- **推測禁止**: サブエージェントも自分も、ファイルを実際に読んで確認した事実のみを乖離として報告する
- **git管理外ファイルに注意**: 生成物（`__pycache__`、`.venv` 等）はディレクトリ構造チェックの対象外とする

## チェックリスト

実行完了前に必ず確認:

- [ ] `docs/` の正式版ドキュメントを全て列挙したか？（ideas/ を除外したか？）
- [ ] 存在するドキュメント全てに対応するチェックを実行したか？
- [ ] 各乖離に「ドキュメントの記載箇所」と「コードの箇所（file:line）」の両方を明記したか？
- [ ] 修正方向（ドキュメント更新 or コード修正）を全件で推奨したか？
- [ ] 未実装機能を乖離として誤報告していないか？
- [ ] レポートのみ提示し、承認なしに修正していないか？
