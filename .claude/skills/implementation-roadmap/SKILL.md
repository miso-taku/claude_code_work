# 実装ロードマップスキル

このスキルは、プロダクト全体の実装順序を機能レベルで定義する「実装ロードマップ」を作成・更新するための詳細ガイドです。

## このドキュメントの役割（最重要）

実装ロードマップは**粗い全体地図**であり、詳細なタスクリストではありません。

- **書くこと**: 機能一覧（`/add-feature` の実行単位）、フェーズ分け、機能間の依存関係、進捗状態
- **書かないこと**: 個々の実装タスク（テスト作成、クラス実装など）。詳細タスクは `/add-feature` 実行時に `.steering/[日付]-[機能名]/tasklist.md` として生成される

この役割分担により、実装が進んでも陳腐化しにくい「北極星」ドキュメントとして機能します。

## 前提条件

実装ロードマップの作成を開始する前に、以下のドキュメントが存在することを確認してください:

1. `docs/product-requirements.md` (PRD) — 機能一覧と優先度の源泉
2. `docs/functional-design.md` (機能設計書) — 機能の分割単位の源泉
3. `docs/architecture.md` (アーキテクチャ設計書) — 技術的依存関係の源泉

## 既存ドキュメントの優先順位

**重要**: `docs/implementation-roadmap.md` に既存のロードマップがある場合:

1. **既存のロードマップ (`docs/implementation-roadmap.md`)** - 最優先
   - 完了済み機能の記録（チェック状態、ステアリングへのリンク）は絶対に消さない
2. **このスキルのガイド** - 参考資料

**新規作成時**: このスキルのテンプレートとガイドを参照
**更新時**: 既存の進捗記録を維持しながら、機能の追加・並び替え・状態更新のみ行う

## 機能分割の原則

各行は「`/add-feature` 1回で完了できる単位」に分割します:

- 1機能 = 1ステアリングディレクトリ = 数時間〜数日で完了する粒度
- 大きすぎる場合は「基盤」と「拡張」に分割する（例: 「認証基盤」と「パスワードリセット」)
- 分割の妥当性は functional-design.md の機能一覧と突き合わせて確認する

## フェーズ分けの原則

- **Phase 1 (MVP)**: それが無いとプロダクトが成立しない最小機能群
- **Phase 2 以降**: 価値の高い順。ただし技術的依存が優先順位に勝る（依存元が先）
- 各フェーズには「このフェーズ完了時に何ができるようになるか」というゴールを1文で書く

## 依存関係の原則

- 依存は機能IDで明記する（例: `F-03` は `F-01` に依存)
- 依存が無い機能同士は並行実装可能であることを意味する
- 循環依存を発見した場合は機能分割を見直す

## 更新タイミング

| タイミング | 更新内容 |
|-----------|---------|
| `/add-feature` 完了時 | 該当機能を `[x]` にし、ステアリングディレクトリへのリンクを記載 |
| 機能の追加・変更時 | PRD/機能設計書の更新と同時に行を追加・修正 |
| フェーズ完了時 | フェーズの完了日を記載 |

## 出力先

```
docs/implementation-roadmap.md
```
